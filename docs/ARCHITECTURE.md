# Архитектура Контрольки

Этот документ — короткая экскурсия для разработчика, который впервые открыл
проект и пока видит перед собой много Qt, фотографий и подозрительно большой
`app.py`. Всё нормально: основные тяжёлые задачи уже разнесены по модулям, а
`app.py` в первую очередь связывает их с пользовательскими сценариями.

## С чего начать

Точка входа — `rawww:main`. Ленивый вызов из `src/rawww/__init__.py` передаёт
управление `rawww.app.main()`, где создаются `QApplication`, главное окно и
механизм единственного экземпляра.

Основная иерархия интерфейса выглядит так:

```text
MainWindow
├── TransferManager — общая очередь копирования и перемещения
└── Workspace (по одному на вкладку)
    ├── PhotoGrid — сетка папок и фотографий
    ├── TransferQueuePanel — представление общей очереди ниже избранного
    ├── FullView — полноэкранный просмотр
    │   ├── FullImageView — отрисовка, 100 % и панорамирование
    │   └── ViewerStrip — нижняя лента и раскрытые серии
    ├── FolderCache — SQLite-кэш текущей папки
    ├── DecodeScheduler — очереди декодирования
    ├── MetadataPipeline — фоновый EXIF
    └── AiPipeline — лица, эмбеддинги и анализ фокуса
```

`MainWindow` управляет вкладками и общей оболочкой. Содержимое конкретной папки
принадлежит `Workspace`: именно он знает текущий путь, выделение, фильтры и
поколение фоновых запросов.

Повторный запуск из Проводника передаёт путь первой копии через локальный сокет.
На Windows второй процесс также передаёт право foreground-активации: первая
копия нативно восстанавливает свёрнутый HWND и только затем получает фокус.

## Как открывается папка

1. `Workspace.load_directory()` создаёт новый идентификатор запроса и запускает
   сканирование вне UI-потока.
2. Результат принимается только в том случае, если запрос всё ещё относится к
   текущей папке. Пользователь может переключиться быстрее диска — это не баг,
   это его законное право.
3. Открывается `FolderCache`, из SQLite восстанавливаются превью, EXIF, метки и
   результаты AI.
4. `PhotoGrid` получает элементы небольшими порциями, чтобы event loop Qt не
   превращался в очередь ожидания.
5. `DecodeScheduler` сначала обслуживает видимые карточки и открытый кадр, затем
   постепенно догружает остальную папку.

## Потоки и процессы

Главный поток принадлежит Qt. Виджеты и большинство Qt-объектов нельзя безопасно
трогать из фоновых потоков.

- Потоки используются для ожидания диска, SQLite, сети и ExifTool.
- Процессы используются для тяжёлого декодирования и части AI-задач.
- На Windows главный процесс помещается в Job Object с
  `JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE`; на Linux и macOS приложение работает в
  собственной группе процессов, за которой следит минимальный watchdog.
  Поэтому при штатном или аварийном выходе завершается всё дерево приложения:
  декодеры, AI/EXIF-воркеры и запущенный ими ExifTool не могут остаться сиротами.
- Результаты возвращаются в UI через сигналы `DecodeBridge` и аналогичные
  QObject-мосты.
- Каждый обратный вызов проверяет актуальную папку и поколение запроса. Без этой
  проверки старое превью однажды обязательно приедет в новую вкладку — обычно
  во время демонстрации заказчику.

При закрытии вкладки сначала выставляется признак завершения, затем гасятся
очереди. Такой порядок не даёт позднему callback отправить работу в уже закрытый
executor.

Закрытие фоновых очередей двухфазное. Вкладка немедленно запрещает новую работу,
отменяет ещё не начатые производные задачи и передаёт исполнителей в общий
реестр `task_lifecycle`. Уже начатое декодирование заканчивается штатно. При
окончательном закрытии главного окна приложение дожидается всего реестра,
включая исполнителей от ранее открытых папок и уже закрытых вкладок. XMP и
финальный сброс SQLite-кэша не отменяются, потому что они фиксируют изменения
пользователя, а не только восстанавливаемые результаты анализа.

Активная вкладка имеет приоритет по ресурсам. При скрытии вкладки её очередь
превью и незавершённый AI останавливаются; при возвращении недостающие результаты
достраиваются по данным кэша. При смене папки сканирование, загрузка кэша и
декодирование сразу получают свежих исполнителей, поэтому не ждут очереди старой
папки. Результат декодирования принимается только при совпадении пути и поколения
запроса — повторное открытие той же папки тоже не делает старый callback
актуальным.

## Файловые операции

`transfer_queue.py` отделяет долгоживущие копирование и перемещение от вкладки,
из которой они были запрошены. Единственный `TransferManager` принадлежит
`MainWindow`; панели всех `Workspace` подписаны на одно состояние. Поэтому две
операции, добавленные из разных вкладок, по умолчанию выполняются строго по
очереди. Настройка может разрешить до трёх параллельных задач.

Диалог быстрого переноса только выбирает цель и согласует корневой конфликт
имени. По умолчанию конфликт получает свободное имя автоматически. Рабочий
поток сначала считает файлы и байты, затем пишет каждый корневой объект во
временную цель рядом с итоговой и публикует её атомарным переименованием. При
паузе или отмене временная цель удаляется, а исходник перемещения удаляется
только после успешной публикации копии.

Импорт карт памяти координирует `MainWindow`: `Workspace` показывает кнопку
для уже найденных съёмных томов и передаёт выбранные карты только как намерение.
`card_import.py` в отдельном потоке обходит карты, отбрасывает служебные файлы
по расширению и формирует явные `TransferEntry`. Поэтому существующая очередь
сохраняет единые прогресс, отмену, атомарную публикацию и удаление источника.
После полностью успешной основной задачи главное окно ставит отдельную задачу
резервного копирования с теми же итоговыми именами и структурой.

Копирование на всех платформах использует один внутренний путь с
переиспользуемым буфером 16 МБ. Это даёт одинаковые паузу, отмену и точный
64-битный прогресс для больших RAW без платформенных callback. Перемещение
внутри одного тома остаётся системным `rename` и не превращается в лишнее копирование.
Виджеты получают прогресс через сигналы Qt и не читаются рабочими потоками.
Если окно закрывается с незавершённой очередью, пользователь сначала явно
подтверждает отмену.

## Изображения и кэш

`imaging.py` декодирует RAW и обычные изображения в простую структуру
`PixelImage`. В фоне Qt-виджеты не создаются; преобразование в `QImage`
происходит ближе к интерфейсу.

Кэш состоит из двух уровней:

- `DecodeCache` — ограниченный LRU в оперативной памяти для недавно показанных
  кадров и миниатюр;
- `FolderCache` — SQLite-база для каждой папки с превью, EXIF, пользовательскими
  метками, ShotSync-связями и результатами AI.

Всё содержимое `FolderCache` считается производным. Повреждённую базу безопаснее
пересоздать из фотографий, чем героически спасать из неё половину миниатюры.

Готовая миниатюра сначала публикуется в интерфейсе, а затем попадает в
последовательную очередь записи `FolderCache`. Очередь коротко объединяет
результаты в пакет: JPEG-кодирование и SQLite-транзакция не блокируют Qt, а при
смене папки пакет ставится перед закрытием соответствующей базы.

## Метаданные, XMP и AI

`exif.py` держит долгоживущий ExifTool в режиме `-stay_open`: запуск отдельного
процесса на каждый кадр был бы очень надёжным способом проверить терпение
пользователя.

`xmp.py` читает и обновляет sidecar-файлы с рейтингом, цветом, комментарием,
ключевыми словами и областями лиц. Неизвестные XML-узлы, включая настройки
проявки внешнего редактора, сохраняются. Запись выполняется атомарно через
временный файл и требует совпадения отпечатка с прочитанной версией: внезапное
внешнее изменение превращается в повторное чтение, а не в потерю данных.

`FolderCache.xmp_state` хранит отпечаток и последний синхронизированный снимок
управляемых полей. У ручных операций однозначный приоритет: чтение принимает
рейтинг, метки и текст из sidecar, запись — из состояния `Workspace`. Поэтому
отдельного пользовательского состояния конфликта нет. Состояние привязано к
sidecar, поэтому RAW и JPEG с одинаковым stem образуют одну XMP-пару и не
соревнуются за запись.
Наблюдатель папки сначала перечитывает XMP; полный повторный scan запускается
только при изменении набора фотографий.

XMP не участвует в критическом пути показа превью. Обход каталога, XML, digest
и атомарная запись выполняются в отдельной последовательной очереди. Повторная
проверка сначала сравнивает `size/mtime` и разбирает только изменившиеся
sidecar-файлы; отсутствующие XMP не открываются по одному. Большая очередь
подготавливается порциями через event loop, а импорт в SQLite объединяется в
одну фоновую транзакцию. После внешней правки обновляются данные и прямоугольник
конкретной карточки. Полная перестройка списка допустима только когда изменилось
поле активного фильтра, сортировки или поиска.

`ai.py` управляет пакетами анализа, а `face_analysis.py` содержит инференс
моделей лиц. Модели загружаются лениво. Эмбеддинги и лица сохраняются в кэш и
повторно используются интерфейсом и ShotSync.

После детекции для каждого лица `face_analysis.py` одним прогоном модели разметки
106 точек `2d106det` (та же buffalo-семья insightface, что детектор и
распознавание) получает контур век. По контуру геометрически считается eye aspect
ratio каждого глаза; в `eyes_open` кладётся `max` двух глаз, поэтому «закрыто»
означает закрытые оба глаза, а подмигивание кадр не бракует. Геометрия устойчивее
к контровому свету, чем вероятность отдельной сети-классификатора. Значение
`eyes_open` кладётся в ту же JSON-запись лица рядом с `bbox` и `embedding`,
кэшируется как остальной анализ и остаётся необязательным: старые записи без него
и лица без состояния глаз просто игнорируются фильтром. Порог закрытия
(`EYES_OPEN_THRESHOLD`) подобран на кадрах. Правило брака по глазам
(`Workspace._eyes_closed`) бракует кадр, если закрыто крупнейшее лицо, либо когда
лиц с известным состоянием глаз не больше трёх и закрыто хотя бы одно.
UI-кнопка-переключатель «Закрытые глаза» стоит рядом с фильтром крупности плана.

`focus.py` ищет брак по фокусу и смазу без отдельной модели: это классический CV
на numpy/Pillow, поэтому не тянет ещё один ONNX-вес и считается тем же фоновым
`analyze_focus_batch` по уже подготовленному источнику ~640px, что и эмбеддинги с
лицами. Ключевое решение — резкость меряется **порегионно** (сетка 16×16), а не
по всему кадру: иначе малая ГРИП (объект чёткий, фон в боке) бракуется наравне с
промахом фокуса. Кадр считается резким, если резок хотя бы один значимый регион;
брак — когда размыт даже самый чёткий. Резкость региона оценивается методом
повторного размытия (Crete, 2007): доля перепадов яркости, теряемых при лёгком
доразмытии, — мера относительная и устойчивая к контенту и шуму. Горизонталь и
вертикаль считаются отдельно: за регион отвечает худшее направление, а сильная
разница направлений по всему кадру выдаёт смаз (`motion`) против расфокуса
(`defocus`). `focus.py` умеет мерить резкость и внутри рамки лица (фронт/бэк-фокус),
но пайплайн первой версии считает фокус независимо от лиц, чтобы не связывать два
параллельных этапа порядком выполнения. Результат кладётся JSON-строкой в таблицу
`focus_analysis` рядом с `image_embeddings` и `face_analysis`, кэшируется и
инвалидируется по имени, размеру и mtime файла. Правило брака (`focus_is_defect`)
и UI-кнопка-переключатель «Не в фокусе / смаз» повторяют образец фильтра по
глазам. По замеру `ai_benchmark` этап добавляет единицы миллисекунд на кадр —
проценты от полного AI-конвейера.

`face_search.py` автоматически строит из выбранного embedding два взвешенных
ракурсных sub-center. Три ограниченных уточнения находят профиль и анфас одного
человека, но найденные лица не образуют неограниченную транзитивную очередь.
Качество детекции и размер рамки влияют на центры, а в одном кадре остаётся только
лучшее лицо искомого человека. Поэтому клик по разным ракурсам даёт устойчивую
выдачу без ручного наполнения набора эталонов. Поиск выполняется в отдельном потоке:
главный цикл Qt продолжает рисовать бесконечный индикатор.
SQLite читается только при загрузке `photo_details`: первый запрос строит отдельный
плотный индекс `float32` по текущему снимку, а следующие запросы папки переиспользуют
его. Сравнения идут ограниченными блоками и с лимитом потоков BLAS, поэтому временная
память не растёт вместе с размером найденной группы и не забирает все ядра у Qt.

При входе в полный просмотр готовая миниатюра выбранной карточки показывается
до дисковых операций и построения навигации. Нижняя лента создаёт только одну
страницу путей, хотя стрелочная навигация продолжает работать по всей папке:
тысячи невидимых `QListWidgetItem` не должны задерживать первый кадр.

## ShotSync

Интеграция разделена по обязанностям:

- `shotsync_client.py` — асинхронный HTTP API;
- `shotsync_socket.py` — один WebSocket на всё приложение;
- `shotsync_hub.py` — общий координатор вкладок, сокета и приёма файлов;
- `shotsync_receiver.py` — получение оригиналов в реальном времени;
- `shotsync_selection.py` — локальная копия съёмки и очередь меток;
- `shotsync_upload.py` — подготовка превью и отправка папки. Кодирование идёт
  «окном», чтобы не держать всю папку в памяти; сбой файла не рушит отправку:
  транспортные ошибки повторяются с растущей паузой, безнадёжный кадр помечается
  неудачным. Соответствия имя→`photo_id` пишутся в `FolderCache` по мере успеха,
  поэтому повторный запуск той же папки переиспользует съёмку и догружает только
  недостающее вместо создания дубля;
- `shotsync_panel.py` и `shotsync_login.py` — только интерфейс.

Сетевой слой не должен напрямую менять виджеты. Он сообщает о результате
сигналом, а `Workspace` решает, относится ли событие к открытой папке.

## Где размещать новый код

- Отрисовка и обработка событий конкретного виджета — рядом с этим виджетом.
- Декодирование, кэш, сеть и преобразование данных — в отдельном модуле, без
  зависимости от `Workspace`.
- Координация нескольких подсистем в пользовательском сценарии — в `Workspace`.
- Переиспользуемый маленький виджет — в `widgets.py`; большой диалог — в
  `dialogs.py`.
- Чистая функция заслуживает отдельного теста. Она обычно дешевле ещё одного
  вечера с отладчиком.

## Важные договорённости

- Не блокировать Qt event loop диском, сетью, EXIF, AI или большим циклом.
- Не передавать `QPixmap` между процессами; для обмена используются байты и
  простые dataclass-структуры.
- Любой фоновый результат проверять на актуальность перед изменением UI.
- Не считать кэш источником пользовательских файлов.
- Не выполнять файловые операции внутри представлений: виджет посылает сигнал,
  координатор принимает решение.
- Новое неочевидное поведение сопровождать комментарием о причине, а не
  пересказом соседней строки.

## Проверки

Основной набор тестов запускается так:

```powershell
uv run python -m unittest discover -s tests -v
```

После изменений в асинхронном коде отдельно полезно проверить переключение папок,
закрытие вкладки во время загрузки и быстрый скролл большой директории. Именно
там особенно любят жить результаты из прошлого.
